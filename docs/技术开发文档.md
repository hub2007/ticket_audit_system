## 1. 系统概述

### 1.1 我们要解决什么问题

高校网络信息中心每天都会收到大量硬件设备故障报修工单。但师生填写的工单普遍存在三个问题：

| 问题 | 举例 |
|------|------|
| **口语化严重** | "服务器连不上了"、"电脑卡死了" |
| **关键信息缺失** | 忘了写位置、没留电话 |
| **术语不统一** | 有人说"主机"，有人说"服务器"，其实是一回事 |

目前全靠运维老师逐条阅读、凭经验判断，遇到信息不全的还得打电话追问，效率很低。

### 1.2 我们要做什么

做一个**能自动检查工单质量的网页系统**，它能：

1. **自动打分** — 从7个方面给工单评分，70分算合格
2. **给出整改建议** — 哪里不好、怎么改，直接告诉填单人
3. **知识图谱联动** — 提到某个设备，自动显示它常见的故障有哪些
4. **智能体深度分析** — 两套AI分析模式，一套离线可用，一套联网深度推理
5. **合格入库、不合格暂存** — 合格的直接入库，不合格的暂存起来，改好了再提交

### 1.3 核心功能一览

| 编号 | 功能 | 一句话说明 |
|------|------|-----------|
| F1 | 工单录入 | 网页表单，填写设备、故障、位置、电话 |
| F2 | 自动校核评分 | 七层面检查，100分制，70分合格 |
| F3 | 整改建议 | 逐条告诉用户哪里扣分了、怎么改 |
| F4 | 知识图谱展示 | 网页上画出设备-故障-场景的关系网，可点击互动 |
| F5 | 知识图谱提示 | 检测到设备名后，自动弹出该设备常见故障 |
| F6 | 合格入库 | ≥70分自动存入"库存"列表 |
| F7 | 不合格暂存 | <70分自动存入"暂存区"，可调出来修改重交 |
| F8 | 本地智能体 | 离线运行，自动规划检查顺序，给出判断报告 |
| F9 | ReAct云智能体 | 联网调用DeepSeek大模型，多轮推理深度分析 |
| F10 | 库存管理 | 查看、删除、导出TXT文件 |

## 2. 系统架构

### 2.1 整体架构（文字描述）

系统分为两大部分：**前端（网页端）** 和 **后端（Python服务端）**，通过HTTP请求通信。

```
用户浏览器（前端）                     Python程序（后端）
─────────────────                     ─────────────────

[工单录入表单]  ──发送工单数据──►  [Flask 接收请求]
                                    │
                                    ├─ [jieba 分词] 把中文句子切成词语
                                    ├─ [术语匹配] 检查用词是否规范
                                    ├─ [规则引擎] 七层面逐项打分
                                    ├─ [知识图谱] 查询设备-故障关联
                                    └─ [智能体] 决策分析（可选）
                                        │
[结果展示页面]  ◄──返回JSON结果───  ──┘
  ├─ 评分卡片
  ├─ 扣分详情
  ├─ 整改建议
  ├─ 图谱提示
  └─ 智能体报告

[localStorage 本地存储]
  ├─ 已入库工单列表
  └─ 暂存区工单列表
```

### 2.2 架构特点

- **前后端分离**：前端只管界面和交互，后端只管数据处理和打分，互不干扰
- **无需数据库**：数据用JSON文件和浏览器localStorage存储，零部署成本
- **离线也能用**：本地智能体和规则引擎不依赖网络，断网也能跑
- **学习友好**：每个模块功能单一、边界清晰，便于分工学习

### 2.3 一次完整的校核过程

```
1. 用户在网页表单填写：设备=服务器、故障=无法开机、位置=数据中心、电话=138xxxx
2. 点击"提交校核"，前端把数据POST到后端 /api/audit
3. 后端收数据 → jieba分词 → 术语库匹配 → 七层面打分 → 图谱查关联
4. 后端返回JSON结果：总分78、合格、哪里扣了分、怎么改
5. 前端收到结果 → 渲染评分卡片 → 展示扣分详情和建议
6. 78分≥70 → 自动存入localStorage"库存"列表
7. (可选)用户点击"本地智能体分析"→ 后端决策树分析 → 返回检查路径报告
8. (可选)用户点击"ReAct分析"→ 后端调DeepSeek API多轮推理 → 返回深度报告
```

## 4. 知识图谱设计

### 4.1 什么是知识图谱（通俗理解）

把各种东西和它们之间的关系画成一张网：

- **节点（圆圈）**：代表一个东西，比如"服务器"、"网络故障"、"数据中心"
- **连线**：代表两个东西有关系，比如"服务器"——"网络故障"（服务器可能出现网络故障）
- **颜色区分**：蓝=设备、红=故障、绿=位置

这张网就是"知识图谱"。工单里提到某个设备，系统就去这张网里查它相关的故障有哪些。

### 4.2 三类实体

#### 设备资产（8种，蓝色节点）

| 编号 | 设备名 | 俗称/别称 |
|------|--------|-----------|
| D01 | 服务器 | 主机、机架式服务器 |
| D02 | 交换机 | 网络交换机、核心交换机 |
| D03 | 路由器 | 无线路由器、边界路由器 |
| D04 | 防火墙 | 安全网关 |
| D05 | UPS | 不间断电源、备用电源 |
| D06 | 存储设备 | 磁盘阵列、NAS |
| D07 | 打印机 | 激光打印机、多功能一体机 |
| D08 | 办公电脑 | 台式机、工作站 |

#### 故障类型（8种，红色节点）

| 编号 | 故障分类 | 具体表现 |
|------|---------|---------|
| F01 | 硬件故障 | 无法开机、硬盘坏、内存坏、电源坏、风扇不转 |
| F02 | 网络故障 | 上不了网、网速慢、断网、丢包、IP冲突 |
| F03 | 系统故障 | 蓝屏、死机、系统崩溃、反复重启 |
| F04 | 外设故障 | 打印机不打、显示器黑屏、键盘鼠标不灵 |
| F05 | 软件故障 | 装不上、闪退、报错、许可证过期 |
| F06 | 存储故障 | 磁盘满了、读写失败、RAID降级 |
| F07 | 电源故障 | 不通电、电池没电、电压不稳 |
| F08 | 散热故障 | 过热、风扇不转、温度报警 |

#### 运维场景（6种，绿色节点）

| 编号 | 场景 | 包含区域 |
|------|------|---------|
| L01 | 教学楼 | J1、J2、J3、教学楼A、教学楼B |
| L02 | 实验楼 | S1、S2、实验楼A、实验楼B |
| L03 | 图书馆 | 图书馆、图信大楼 |
| L04 | 行政楼 | 行政楼、办公楼 |
| L05 | 数据中心 | 数据中心、机房、网络中心 |
| L06 | 学生宿舍 | 宿舍1-5号楼 |

### 4.3 设备-故障关联关系

这是知识图谱中最核心的关系——哪个设备可能出哪种故障：

| 设备 | 可能发生的故障 | 最常出现的场景 |
|------|---------------|---------------|
| 服务器 | 硬件故障、系统故障、存储故障、散热故障 | 数据中心 |
| 交换机 | 网络故障、硬件故障、电源故障 | 数据中心、教学楼 |
| 路由器 | 网络故障、电源故障、硬件故障 | 数据中心、教学楼 |
| 防火墙 | 网络故障、系统故障 | 数据中心 |
| UPS | 电源故障、硬件故障 | 数据中心 |
| 存储设备 | 存储故障、硬件故障 | 数据中心 |
| 打印机 | 外设故障、网络故障、软件故障 | 教学楼、行政楼、图书馆 |
| 办公电脑 | 系统故障、硬件故障、软件故障、外设故障 | 教学楼、行政楼、实验楼 |

**校核时怎么用**：工单说"服务器出现网络故障"→ 查图谱 → 服务器和网络故障之间有关联 → 合理，加分。工单说"打印机出现蓝屏"→ 查图谱 → 打印机和系统故障（蓝屏）没有直接关联 → 可能填错了，扣分+提醒。

### 4.4 图谱数据存储格式

用 JSON 文件存储，结构简单清晰：

```
kg.json
├── entities（实体列表）
│   ├── devices：8个设备节点（id、名称、别名）
│   ├── faults：8个故障节点（id、名称、典型症状）
│   └── locations：6个场景节点（id、名称、子区域）
└── relations（关系列表）
    └── 每条关系：谁→谁→什么关系→关联强度(0-1)
```

### 4.5 知识图谱可视化方案

使用网页 **Canvas（画布）** 来画，就像在网页上放了一张画纸，可以用代码在上面画圆、写字、画线。

**画法思路**：
1. 画圆形节点：不同颜色代表不同类型（蓝/红/绿）
2. 画文字标签：节点旁边写名字
3. 画连线：有关联的节点之间画灰色线
4. 点击交互：点到哪个节点，它就高亮，旁边显示详细信息

**提示**：不用一开始就追求完美布局，先用简单的位置排列（比如三类各排一列），后面再慢慢调好看。

### 4.6 图谱查询功能

系统需要能查询图谱的几种方法（后端提供，前端调用）：

| 查询场景 | 输入 | 输出 |

## 5. 语义分析与校核规则引擎

### 5.1 四层词库体系

为了让系统能"理解"工单里写的是什么，我们需要建立四层词库：

```
第四层：标准术语转换表
  作用：口语→标准话的翻译（"连不上"→"网络中断"）
  文件：data/fuzzy_map.json

第三层：模糊表述词库
  作用：收录常见的非标准说法
  文件：data/fuzzy_map.json（同文件）

第二层：故障术语库
  作用：收录标准故障分类及典型表现
  文件：data/fault_terms.json

第一层：设备术语库
  作用：收录标准设备名及别名
  文件：data/device_terms.json
    +
位置术语库
  作用：收录标准位置名及别名
  文件：data/location_terms.json
```

**词库之间的关系**：

- 用户写"主机"→ 查设备术语库 → 知道"主机"就是"服务器"（别名匹配）
- 用户写"连不上"→ 查模糊表述词库 → 知道该转换为"无法连接"（口语标准化）
- "无法连接"→ 查故障术语库 → 知道这属于"网络故障"（故障归类）

### 5.2 中文分词与语义理解

#### 分词是什么

把一句话切成一个个有意义的词。比如：

```
"数据中心服务器无法开机了"
     ↓ jieba分词
["数据中心", "服务器", "无法", "开机", "了"]
```

分词后系统就能识别出："数据中心"=位置、"服务器"=设备、"无法开机"=故障。

#### 语义相似度计算

用**词语重叠率**来判断两段文字是否在说同一件事：

```
"服务器连不上网" 和 "服务器无法上网"
         ↓ 分别分词
["服务器","连不上","网"]  和  ["服务器","无法","上网"]
         ↓ 看有多少词重叠
共同词：1个（"服务器"），总不同词：5个
相似度 = 2×1/5 = 40%（偏低，因为"连不上"和"无法上网"没对上）
         ↓ 但如果先经模糊词库转换
["服务器","无法上网","网"] 和 ["服务器","无法","上网"]
         ↓ 再算
相似度提高
```

> **说明**：这个算法不复杂，核心就是"切词→看重叠→算比例"，大一的数学基础完全够用。

#### 匹配流程

```
工单中的一段文字
    │
    ├─ 第1步：精确匹配 — 直接查词库里有没有一模一样的词
    │   找到 → 直接返回标准名
    │   没找到 ↓
    │
    ├─ 第2步：别名匹配 — 在词库的所有别名里找
    │   找到 → 返回对应的标准名
    │   没找到 ↓
    │
    └─ 第3步：相似度模糊匹配 — 一个词一个词地算相似度
        取相似度最高的，超过60%就采纳
        低于60% → 标记为"无法识别"
```

### 5.3 七层面规则引擎

这是整个系统最核心的打分机制。就像老师改卷子，每道题有分值，系统从7个方面给工单打分。

#### 评分总表

| 序号 | 检查层面 | 满分 | 查什么 | 扣分逻辑 |
|------|---------|------|--------|---------|
| 1 | 设备完整性 | 20分 | 有没有写设备名？设备名能在词库里找到吗？ | 没写扣20，词库找不到扣5-15 |
| 2 | 故障明确性 | 20分 | 有没有描述故障？描述够不够清楚？ | 没写扣20，太模糊扣5-15 |
| 3 | 位置完整性 | 10分 | 有没有写位置？位置能在词库里找到吗？ | 没写扣10，对不上扣3-5 |
| 4 | 联系方式 | 10分 | 有没有填电话？是11位手机号或座机格式吗？ | 没填扣10，格式错扣5 |
| 5 | 术语规范性 | 15分 | 有没有口语化表述？（如"卡"、"死机了"） | 发现一处口语扣3分，最多扣15 |
| 6 | 知识图谱一致性 | 15分 | 说的设备和故障在知识图谱里有关联吗？ | 没关联扣10，查不到扣5 |
| 7 | 逻辑完整性 | 10分 | 整段话读起来通顺吗？有没有自相矛盾？ | 有矛盾扣5-10 |

**总分100分，70分合格。**

#### 评分等级

| 分数 | 等级 | 含义 | 处理方式 |
|------|------|------|---------|
| 90-100 | A（优秀） | 各方面都很规范 | 直接入库 |
| 80-89 | B（良好） | 有小瑕疵但不影响 | 直接入库 |
| 70-79 | C（合格） | 达标，但有改进空间 | 直接入库（附建议） |
| 60-69 | D（待改进） | 有明显问题 | 进暂存区，修改重交 |
| 0-59 | E（不合格） | 严重缺失 | 进暂存区，必须大改 |

#### 各层面检查逻辑（自然语言描述）

**层面1：设备完整性（20分）**

1. 先看工单里"设备名称"这一栏有没有填
2. 如果没填 → 0分，建议"请填写设备名称"
3. 如果填了 → 拿这个词去设备术语库匹配
4. 精确匹配到了 → 满分20
5. 通过别名匹配到了 → 满分20，备注"已自动识别为xxx"
6. 模糊匹配到了（相似度60%以上）→ 扣3-10分，建议改为标准名
7. 完全匹配不到 → 5分，建议"请确认设备名称是否正确"

**层面2：故障明确性（20分）**

1. 先看工单里"故障描述"有没有填
2. 如果没填 → 0分
3. 如果填了但只有一两个字（如"坏了"）→ 5分，建议"请详细描述故障现象"
4. 如果填了，去故障术语库匹配到了 → 18-20分
5. 如果匹配不到但描述较长 → 10分，说明不够标准但有一定信息量

**层面3：位置完整性（10分）**

1. 没填 → 0分，建议"请填写设备所在位置"
2. 填了，在位置术语库找到了匹配 → 10分
3. 填了，模糊匹配到 → 5-7分
4. 填了，匹配不到 → 3分（至少有信息）

**层面4：联系方式（10分）**

1. 没填 → 0分
2. 填了，格式是11位且1开头 → 10分（手机号）
3. 填了，格式是0开头+连字符+数字 → 10分（座机号）
4. 填了但格式不对 → 5分，建议"请填写正确的手机号或座机号"

**层面5：术语规范性（15分）**

1. 把工单全文做分词
2. 每个词去模糊词库（fuzzy_map）里查
3. 每发现一个口语词，扣3分
4. 提示："xxx建议改为标准术语yyy"
5. 扣到0分为止（15分满分最多发现5处口语词）

**层面6：知识图谱一致性（15分）**

1. 先确认设备名和故障分类是否已识别成功
2. 如果设备或故障未识别 → 5分，无法做一致性检查
3. 如果都识别了 → 去知识图谱查设备-故障关联
4. 有关联 → 满分15分
5. 没有关联 → 5分，提示"该设备通常不会出现此类故障，请核实"
6. 关联存在但权重低 → 10分，提示"该关联较少见，请确认"

**层面7：逻辑完整性（10分）**

1. 检查整体信息是否自洽
2. 比如：设备说"打印机"，位置说"数据中心"——打印机一般不在数据中心，可能有问题
3. 比如：故障说"无法开机"，但设备栏写的是"软件"——软件不会"开机"，逻辑矛盾
4. 没问题 → 10分
5. 有轻微矛盾 → 5-7分
6. 有明显矛盾 → 0-4分

### 5.4 规则引擎怎么实现（思路说明）

核心就是：**if-else 条件判断**。这也是立项书里反复提到的"Python条件判断"。

基本结构：
1. 接收工单数据（设备、故障、位置、联系方式）
2. 先分词、做术语匹配（调用jieba和词库）
3. 然后依次执行7个检查函数，每个返回{得分, 说明, 建议}
4. 把7个结果加起来得总分
5. 判断 ≥70 还是 <70，给出等级
6. 打包成JSON返回给前端

> **学习要点**：整个规则引擎就是大量的 if-else + 字符串操作 + 查字典。大一上学期学的条件判断、循环、字符串处理，在这里全部用上了。

## 6. 双智能体决策系统

### 6.1 为什么需要智能体

规则引擎能处理大部分情况，但它是**死**的——检查顺序是写死的，不能根据内容灵活调整。

智能体的作用是：**让系统能像人一样思考**——看到工单内容后，自己判断"这个工单最缺什么信息，应该先查哪里"。

### 6.2 第一种：本地规则型智能体（离线运行）

#### 通俗理解

就像一个有经验的运维老师傅，看到工单后会自然做出判断：
- "连设备名称都没写？那我先盯着设备这一项查。"
- "设备写了，但故障描述特别模糊？那我重点检查故障明确性。"
- "四要素都齐全？那我再看深一点——设备跟故障在知识图谱里对得上吗？"

#### 工作流程（四步循环）

```
感知 ──► 决策 ──► 执行 ──► 反馈
 │         │         │         │
 │         │         │         └─ 把报告展示给用户
 │         │         └─ 按照选好的路径逐项检查
 │         └─ 根据缺失情况选一条检查路径
 └─ 读工单内容，搞清楚缺了什么信息
```

#### 决策逻辑

系统读完工单后，按以下优先级排序检查顺序：

| 发现什么情况 | 优先检查什么 |
|-------------|-------------|
| 设备名为空 | 优先查"设备完整性" |
| 故障描述为空 | 优先查"故障明确性" |
| 位置为空 | 优先查"位置完整性" |
| 联系方式为空 | 优先查"联系方式" |
| 有口语化表述 | 优先做"术语规范性"检查 |
| 四要素都齐全 | 优先查"知识图谱一致性"（深度检查） |

然后在报告里说明："因为发现XX缺失，所以优先检查了XX；之后依次检查了XX、XX……"

#### 实现方式

用 **if-elif-else 决策树** 实现（大一基础编程完全能驾驭），不需要额外依赖任何库。

### 6.3 第二种：ReAct 云端智能体（需联网）

#### 通俗理解

ReAct = **推理(Thought) + 行动(Action) + 观察(Observation)** 循环。

跟本地智能体比，它更聪明——因为背后接入了**大语言模型（DeepSeek）**，能自己"思考"下一步该做什么。

#### 工作流程

```
第1轮：思考"先看看工单里设备名写对没有"
        ↓ 行动：调用"设备完整性检查"工具
        ↓ 观察：工具返回"匹配成功，服务器，20分"
        ↓
第2轮：思考"设备没问题，再看看故障描述清不清楚"
        ↓ 行动：调用"故障明确性检查"工具
        ↓ 观察：工具返回"描述模糊，15分"
        ↓
第3轮：思考"故障不够清楚，查查知识图谱里服务器常见故障有哪些"
        ↓ 行动：调用"图谱查询"工具
        ↓ 观察：返回[硬件故障,系统故障,存储故障,散热故障]
        ↓
... 继续直到7条规则都覆盖或达到5轮上限 ...
        ↓
最终：把所有观察结果汇总，写出一份深度分析报告
```

#### 它可以调用的工具

系统预先定义好几个"工具函数"，大模型决定什么时候调用哪个：

| 工具名 | 功能 |
|--------|------|
| 检查设备完整性 | 检查设备名是否在术语库中 |
| 检查故障明确性 | 检查故障描述是否标准 |
| 查询图谱关联 | 查某设备和某故障在知识图谱里有没有关系 |
| 查询设备常见故障 | 列出某个设备的常见故障有哪些 |
| 检查口语化表述 | 扫描文本中的不标准用词 |
| 验证联系方式 | 检查手机号/座机号格式 |

#### 为什么最多5轮

- 控制API成本（每次调用都要消耗token）
- 7条规则可以在5轮内覆盖完（一轮可以调多个工具）
- 防止大模型无限循环

#### 实现方式

通过Python发送HTTP请求调用DeepSeek API（类似于前端调后端，只不过这里是后端调外部API）。

### 6.4 两种模式对比

| | 本地智能体 | ReAct云智能体 |
|------|-----------|--------------|
| 运行条件 | 不需要网络 | 需要联网 + API Key |
| 智能程度 | 固定决策树 | 大模型自主推理 |
| 推理方式 | 按预设规则走路径 | "思考→行动→观察"循环 |
| 速度 | 极快（毫秒级） | 较慢（等API返回，秒级） |
| 成本 | 免费 | 需消耗API额度 |
| 适合场景 | 日常快速校核 | 疑难工单深入分析 |
| 用户怎么选 | 点"本地分析"按钮 | 点"深度分析"按钮 |

### 7.2 交互说明

#### 填写表单时
- 用户在"设备名称"栏打字时，系统实时检测是否匹配到词库中的设备
- 匹配到后，在旁边显示"已识别：服务器"
- 同时知识图谱区自动高亮对应的设备节点

#### 提交校核后
- 分数用大号数字显示，70以上绿色，70以下红色
- 7个检查项用卡片展示，通过的绿色打勾，扣分的红色打叉
- 每个扣分项下面有小字说明原因和修改建议
- 知识图谱联动提示出现在图谱区旁边

#### 智能体分析
- 默认不自动运行（省时间），用户想看的时候点按钮
- 本地智能体：点击后1秒内出报告
- ReAct智能体：点击后显示"分析中..."，等几秒出报告，同时展示推理过程（第1轮做了什么、第2轮做了什么...）

- 合格工单自动进入"已入库"列表，按时间排序
- 不合格工单进入"暂存区"，每条后面有"修改"按钮
- 点"修改"：表单自动填入原来的内容，用户改完再提交
- 点"删除"：从列表中移除
- 点"导出TXT"：浏览器下载一个文本文件，内容是表格格式

### 7.3 数据在浏览器里怎么存

使用浏览器的 **localStorage**（本地存储），特点：
- 数据存在用户自己的浏览器里
- 关了网页再打开，数据还在
- 不需要服务器和数据库
- 每个浏览器独立，换台电脑看不到之前的

存储两样东西：
1. **工单库存**：所有合格工单的列表
2. **暂存区**：所有不合格工单的列表

## 8. 后端接口设计

### 8.1 接口总览

后端提供6个接口（API），前端通过JavaScript的Fetch调用：

| 编号 | 方法 | 路径 | 用途 |
|------|------|------|------|
| 1 | GET | `/` | 确认后端正在运行 |
| 2 | POST | `/api/audit` | **核心接口**：提交工单做校核打分 |
| 3 | POST | `/api/agent/local` | 本地智能体分析 |
| 4 | POST | `/api/agent/react` | ReAct云端智能体分析 |
| 5 | POST | `/api/kg/query` | 查询知识图谱 |
| 6 | GET | `/api/kg/data` | 获取知识图谱全部数据（给前端画图用） |

### 8.2 核心接口：工单校核

**请求方式**：前端发送一个JSON给后端

```json
{
  "device": "服务器",
  "fault": "无法开机了",
  "location": "数据中心",
  "contact": "13800138000"
}
```

**返回方式**：后端返回一个JSON给前端

```json
{
  "code": 0,
  "msg": "success",
  "data": {
    "total_score": 78,
    "passed": true,
    "grade": "C",
    "details": [
      {
        "layer": "设备完整性",
        "score": 20,
        "max_score": 20,
        "status": "pass",
        "message": "设备名称"服务器"匹配正常"
      },
      {
        "layer": "故障明确性",
        "score": 15,
        "max_score": 20,
        "status": "warn",
        "message": "发现口语化表述"了"，建议改为标准用语"
      }
    ],
    "suggestions": [
      "建议将故障描述改为标准术语"
    ],
    "kg_hints": [
      "服务器常见故障：硬件故障、系统故障、存储故障、散热故障"
    ]
  }
}
```

**返回数据中各字段的含义**：

| 字段 | 含义 |
|------|------|
| code | 0=成功，其他数字=出错 |
| total_score | 总分（0-100） |
| passed | true=合格(≥70)，false=不合格 |
| grade | 等级 A/B/C/D/E |
| details | 7条检查的详细结果，每条含得分、满分、状态(pass/warn/fail)、说明 |
| suggestions | 整改建议列表，告诉用户怎么改 |
| kg_hints | 知识图谱提示，告诉用户这个设备还有哪些常见故障 |

### 8.3 本地智能体接口

请求：和校核接口一样（发工单内容）
返回：多了 `perception`（感知到缺失了什么）、`decision_path`（决策路径说明）、`summary`（智能体总结）

### 8.4 ReAct智能体接口

请求：和校核接口一样
返回：多了 `rounds`（推理了几轮）、`trace`（每轮做了什么）、`final_report`（最终分析报告全文）

### 8.5 状态码约定

| code | 含义 |
|------|------|
| 0 | 一切正常 |
| 1001 | 请求缺少必填字段 |
| 1002 | 字段格式不对 |
| 2001 | 知识图谱查询无结果 |
| 3001 | DeepSeek API调用失败（可能网络问题或API Key没配） |
| 9999 | 服务器内部出错 |

## 9. 工单管理流程

### 9.1 状态流转图

```
                    ┌──────────┐
                    │  用户填写  │
                    │  工单表单  │
                    └────┬─────┘
                         │ 点"提交校核"
                         ▼
                  ┌──────────────┐
                  │  后端校核引擎  │
                  │  七层面打分    │
                  └──────┬───────┘
                         │
              ┌──────────┴──────────┐
              │ score >= 70         │ score < 70
              ▼                     ▼
       ┌──────────┐          ┌──────────┐
       │  合格入库  │          │  暂存区   │
       │ (inventory)│          │ (pending) │
       └──────────┘          └────┬─────┘
                                  │ 用户点"修改"
                                  │ 表单自动回填
                                  │ 修改后点"重新提交"
                                  ▼
                           ┌──────────┐
                           │ 再次校核  │
                           └──────────┘
                           (可能合格入库或继续暂存)
```

### 9.2 相似度防重复

提交新工单时，先跟"已入库"和"暂存区"里的工单比较一遍。如果内容太像（比如跟某条已有工单相似度超过70%），就提醒用户"可能重复，已有相似工单xxx"。

实现方法：用前面说的"词语重叠率"算法，把新工单跟每条已有工单算一遍，取最高的。

## 12. 部署说明

### 12.1 运行环境

| 需要的 | 具体要求 |
|--------|---------|
| Python | 3.8 或更高版本 |
| Python包 | flask、jieba（一行命令安装） |
| 浏览器 | Edge、Chrome最新版 |
| 网络 | 基础功能不需要网络；ReAct智能体需要联网 |

### 12.2 怎么跑起来

```bash
# 第一步：安装 Python 依赖（只需要做一次）
pip install flask jieba

# 第二步：启动后端
cd backend
python app.py
# 看到 "Running on http://127.0.0.1:5000" 就说明跑起来了

# 第三步：打开前端
# 方式一：浏览器直接打开 frontend/index.html
# 方式二：如果跨域问题，用Python起个简单的文件服务器
cd frontend
python -m http.server 8000
# 浏览器打开 http://localhost:8000
```

### 12.3 如果要用 ReAct 智能体

需要在 `config.py` 里填写 DeepSeek API Key（去 DeepSeek 官网注册即可获取）。不填的话只是 ReAct 用不了，其他功能不受影响。

## 附录

### A. 术语通俗解释

| 术语 | 大白话解释 |
|------|-----------|
| 工单 | 记录故障报修的一张表，包含设备名、故障描述、位置、联系方式 |
| 知识图谱 | 一张"关系网"，把设备、故障、位置连在一起 |
| 语义校核 | 让系统"读懂"工单内容，检查写得好不好 |
| 实体 | 知识图谱里的一个"东西"（一台设备、一种故障、一个地点） |
| 关系 | 两个实体之间的连线（"服务器"和"硬件故障"之间有线连着） |
| 分词 | 把一句话切成一个个词语 |
| 规则引擎 | 按照写好的规则一条条打分 |
| 智能体 | 能自己判断"下一步该做什么"的程序 |
| ReAct | "推理→行动→观察"循环，大模型的思考方式 |
| 暂存区 | 不合格工单暂时存放的地方，改好了再提交 |
| localStorage | 浏览器自带的存储功能，关了网页数据也不丢 |
| API | 后端提供给前端调用的功能接口 |
| JSON | 一种通用的数据格式，人和程序都能读 |

### B. 核心文件说明

| 文件 | 现在状态 | 需要做什么 |
|------|---------|-----------|
| `backend/app.py` | 有基础框架 | 添加校核、智能体等路由 |
| `data/kg.json` | 空结构 | 填充8设备+8故障+6场景+40+关系 |
| `data/device_terms.json` | 有6类设备 | 补全到8类，检查别名 |
| `data/fault_terms.json` | 有5类故障 | 补全到8类，检查症状列表 |
| `data/location_terms.json` | 有5类场景 | 补全到6类，检查子区域 |
| `data/fuzzy_map.json` | 有6条映射 | 大量扩充口语化→标准词 |
| `docs/技术开发文档.md` | 本文档 | 随开发进度持续更新 |
